## Context

psi-agent 项目当前有 580 个测试用例，总体覆盖率 89%。但部分关键模块覆盖率不足：

- **AI 服务器模块**：`anthropic_messages/server.py` (65%) 和 `openai_completions/server.py` (78%) 的错误处理路径和启动逻辑未测试
- **Workspace API 模块**：`umount/api.py` (63%) 和 `mount/api.py` (78%) 的边缘情况未覆盖
- **Session 模块**：`cli.py` (74%) 和 `server.py` (76%) 的部分代码路径缺失测试

这些低覆盖率区域主要涉及：
- 服务器启动和关闭逻辑
- 错误处理分支（连接失败、超时、无效输入）
- CLI 参数验证和错误提示
- 边缘情况（空输入、异常状态）

## Goals / Non-Goals

**Goals:**
- 为覆盖率低于 80% 的模块添加测试，达到目标覆盖率
- 覆盖错误处理路径和边缘情况
- 确保新增测试遵循现有测试模式（pytest + async）

**Non-Goals:**
- 不重构现有代码结构
- 不改变现有测试框架或工具
- 不追求 100% 覆盖率（某些路径可能不可测试）

## Decisions

### 决策 1：使用 pytest + pytest-asyncio 模式

**选择**：继续使用现有测试模式（pytest + pytest-asyncio + unittest.mock）

**理由**：
- 与现有测试保持一致
- pytest-asyncio 已配置 AUTO 模式，无需额外装饰器
- unittest.mock 用于模拟外部依赖

**替代方案**：
- 使用 pytest-mock：需要额外依赖，与现有 mock 使用不一致

### 决策 2：测试文件组织

**选择**：新增测试放在对应模块的测试目录下

**理由**：
- 与现有测试结构一致（`tests/<module>/test_<file>.py`）
- 便于定位和维护

### 决策 3：测试策略

**选择**：优先覆盖错误处理和边缘情况

**理由**：
- 正常路径大多已测试
- 错误处理路径是回归风险最高的区域
- 边缘情况测试可验证代码健壮性

## Risks / Trade-offs

**风险 1：某些代码路径可能不可测试**
→ 缓解：标记为 `pragma: no cover` 或使用集成测试

**风险 2：新增测试可能增加 CI 时间**
→ 缓解：保持测试简洁，避免重复测试相同逻辑

**风险 3：过度测试可能降低代码可维护性**
→ 缓解：只测试有意义的行为，不测试实现细节
