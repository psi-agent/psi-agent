## Context

psi-ai 模块包含两个 LLM 提供商适配器（openai_completions 和 anthropic_messages），每个适配器都有对应的 CLAUDE.md 文档。PR #113 对代码进行了改进，引入了新的设计惯例：

1. **流式请求日志惯例**：`_stream_request` 方法现在记录完整请求 body（DEBUG 级别），与非流式请求保持一致的日志粒度
2. **流式完成日志惯例**：Server 的 `_handle_streaming` 方法在流式完成后记录两条日志（DEBUG + INFO），提供更清晰的完成信号
3. **非流式 tool_use 转换**：`translate_anthropic_to_openai` 函数现在正确提取 `tool_use` content blocks 并转换为 OpenAI `tool_calls` 格式

现有文档未记录这些设计惯例，需要更新以确保文档准确反映模块的设计思路和规范。

## Goals / Non-Goals

**Goals:**
- 更新 CLAUDE.md 文档以反映当前的设计思路和惯例
- 记录流式请求日志惯例（设计一致性）
- 记录流式完成日志惯例（生命周期可见性）
- 记录非流式响应中 tool_use 转换的接口规范

**Non-Goals:**
- 不修改任何代码实现
- 不改变 API 或运行时行为
- 不添加新功能

## Decisions

### 1. 文档更新位置

**决定**：在现有 CLAUDE.md 的相关章节补充惯例说明，而非新建章节。

**理由**：
- 流式日志惯例属于"日志粒度"规范的延伸，应放在现有日志规范章节
- tool_use 转换属于协议转换接口的一部分，应放在现有转换逻辑章节
- 保持文档结构连贯性

### 2. 文档粒度

**决定**：记录设计惯例和接口规范，而非具体代码行号。

**理由**：
- CLAUDE.md 的目的是传达设计思路、规范、接口、惯例
- 具体代码行号会随重构失效
- 惯例和规范具有持久价值

## Risks / Trade-offs

**风险**：文档更新可能不完整或有遗漏 → 通过完整阅读代码文件确保覆盖所有变更

**权衡**：文档更新后可能与未来代码变更不同步 → 这是所有文档的固有问题，需在 PR review 时检查文档一致性
