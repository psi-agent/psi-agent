## Context

当前实现使用 `setproctitle` 库在进程启动后修改进程标题，将敏感参数（如 `api_key`、`token`）替换为 `***`。这种方式存在以下问题：

1. **时间窗口漏洞**：进程启动到调用 `mask_sensitive_args` 之间存在毫秒级时间窗口，期间敏感信息在进程列表中可见
2. **依赖开销**：引入了 `setproctitle` 依赖，增加了维护成本
3. **代码复杂性**：需要在每个 CLI 入口手动调用 `mask_sensitive_args`，容易遗漏

## Goals / Non-Goals

**Goals:**
- 删除 `proctitle.py` 模块和 `setproctitle` 依赖
- 简化 CLI 入口代码
- 更新文档，建议用户通过环境变量传递敏感信息

**Non-Goals:**
- 不提供替代的打码方案
- 不修改 CLI 参数解析逻辑

## Decisions

### 1. 直接删除而非替换

**决定**：直接删除 `mask_sensitive_args` 功能，不提供替代方案。

**理由**：
- 任何基于命令行参数打码的方案都存在时间窗口漏洞
- 真正的安全解决方案是避免通过命令行传递敏感信息
- 用户应使用环境变量（如 `API_KEY` 环境变量）或配置文件

**替代方案考虑**：
- 在进程启动更早的阶段打码 → 仍有时间窗口，且增加复杂性
- 使用环境变量替换命令行参数 → 改变用户接口，超出本次范围

### 2. 保留 CLI 参数接口

**决定**：保留现有的命令行参数接口（`--api-key`、`--token` 等），仅删除打码调用。

**理由**：
- 不破坏现有用户工作流
- 用户可以自行选择安全方式传递参数（环境变量或配置文件）

## Risks / Trade-offs

**风险**：删除打码后，敏感信息在进程列表中可见 → 用户应通过环境变量传递敏感信息

**权衡**：简化代码 vs. 进程列表可见性 → 选择简化代码，因为打码本身不可靠
